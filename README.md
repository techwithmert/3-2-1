# 3-2-1 · Common Player Checker

Checker for the party game where two people count 3-2-1, each names a football
club, and someone must instantly name a player who played for **both**. Pick the
two clubs, see every player who wore both shirts (with year ranges), type a name
to check a specific claim, and get a yellow card when the clubs share no players.

Built as a static site — no backend, no API keys. Player data is baked in at
build time from [Wikidata](https://www.wikidata.org).

## Develop

```sh
npm install
npm run dev
```

## Data pipeline

The dataset in `public/data/` is generated by three resumable stages
(all state cached in gitignored `scripts/.cache/`):

```sh
npm run data:clubs      # Stage A: enumerate clubs from footballers' P54 links, + labels
npm run data:players    # Stage B: per-club memberships (P54) — the slow crawl, resumable
npm run data:labels     # Stage B2: label every player found, with language fallback
npm run data:wikipedia  # Stage B3: senior careers from Wikipedia infoboxes (~30 min)
npm run data:build      # Stage C: pure transform, emits public/data/
npm run data:all        # all five
```

Data comes from two sources. **Wikidata (P54)** is the backbone — worldwide, all
eras, CC0. **Wikipedia infoboxes** fill its gaps: every Wikidata statement is
added by hand, so it lags badly on recent transfers, while Wikipedia is edited
within hours (Noa Lang's 2026 loan to Galatasaray was on Wikipedia for weeks
while Wikidata had no record of it). Stage B3 contributes ~334k club spells
Wikidata was missing. Wikidata wins any conflict; Wikipedia only adds spells it
doesn't have and fills in blank years.

To refresh the dataset: `rm -rf scripts/.cache && npm run data:all`, then commit
`public/data/` and redeploy. Queries run against QLever by default (no 60s cap);
`SPARQL_ENDPOINT=wdqs` switches to the Wikidata Query Service. `LIMIT_CLUBS=200
npm run data:players` crawls a sample. Stage B checkpoints per club, so an
interrupted crawl resumes where it left off.

### Data files

- `public/data/clubs.json` — `{ v, generated, clubs: [qid, label, iso2, playerCount, parentQid][] }`,
  loaded eagerly to power the club pickers. `parentQid` is set for reserve/youth
  sides and drives the "matches via the B team" hint.
- `public/data/c/{qid}.json` — `{ v, id, p: [playerQid, name, startYear, endYear, birthYear, loan][] }`,
  fetched on demand per selected club; the app intersects two of these client-side.
  Years/birth are `null` when Wikidata lacks them; multiple stints at one club are
  merged to min-start/max-end, and `loan` is 1 only if every stint there was a loan.

### Extraction notes (learned the hard way)

- **Enumerate clubs from the P54 side**, not from a `P31` class whitelist. Clubs
  typed only as "sports club"/"multisports club" — Boca Juniors, San Lorenzo,
  Karşıyaka — are invisible to a class walk of `Q476028`, which silently cost
  ~5,000 clubs.
- **Don't filter on `P641` (sport).** The values are inconsistent: multisport
  clubs list all 16 sports they run, Greek clubs use the subclass "men's
  association football", Mohun Bagan uses "team sport". Every filter formulation
  deleted real clubs.
- **Loans need no special handling** — Wikidata models a loan as its own `P54`
  statement, so they're included by default. `P1642 = Q2914547` only labels them.
- **Use `p:`/`ps:P54`, never `wdt:P54`** — the truthy form returns best-rank
  statements only, which for Falcao means 1 club instead of 13.
- A handful of clubs exist twice (a multisport parent plus a football section).
  Stage C merges those, but only when the section is marked `Q103229495` — B
  teams and women's teams point at the same parent and must stay separate.
- **Don't merge clubs on rename links either** (`P1365`/`P1366`/`P155`/`P156`).
  They encode succession, mergers and relocations rather than "same club": LR
  Vicenza "replaces" both Vicenza Calcio *and* Bassano Virtus, and the Sangmu
  chain crosses cities. So renamed clubs legitimately appear twice (Vicenza
  Calcio and LR Vicenza), each with its own roster.
- Parse the infobox by walking template params, not line-by-line regex — many
  articles put several params on one line, and pipes appear inside `[[A|B]]`.
  Match param names exactly so `youthclubs1` never reads as `clubs1`.

### Sanity check before shipping a rebuild

Every extraction bug here failed *silently*, producing a plausible dataset that
was quietly missing thousands of entries. Always diff against the shipped
`clubs.json` before committing — confirm the club count moved the way you
expect and that **no club lost players**.

## Deploy

Static output — any static host works. Pushing to `main` deploys to GitHub Pages
via `.github/workflows/deploy.yml`. For Vercel instead: connect the repo,
framework preset "Vite", done — `vercel.json` adds cache headers for `/data/*`.

## Notes

- Club and player notability = has at least one Wikipedia article. National teams
  are excluded by construction (the crawl is club-driven).
- Wikidata coverage is imperfect, especially for older eras and smaller leagues —
  a missing player is a data gap, not a rule of the game.
